In Assignment 2, I implemented the ETL pipeline using a mostly procedural design. The logic for reading, transforming, and writing data were written in a small number of classes, and responsibilities were not clearly separated. While the program functioned correctly, it did not fully demonstrate object-oriented design principles. For Assignment 3, I redesigned the solution using object-oriented decomposition and separated responsibilities into different Java classes: Product, ProductTransformer, CsvReader, CsvWriter, ETLPipeline, and Main. The Product file models each product as an object and holds the data for these products. The ProductTransformer file contained business rules (discounts, price range, rounding, etc.). The CsvReader file read and parsed the input data from products.csv. The CsvWriter file wrote the transformed data to transformed_products.csv. The ETLPipeline file coordinated the entire workflow between the other classes, and the Main file was used to run the application. 

Assignment 3 is more object-oriented because it introduces a clearer separation between data, behavior, and system coordination. The Product class models a real-world object, and makes the program more aligned with object-oriented principles by modeling real-world concepts directly in code. Instead of manipulating product information through scattered variables, the Product class stores its attributes privately and uses controlled methods to modify them. This improves data protection and makes the code more structured and maintainable. Reading input, applying transformations, writing output, and coordinating the workflow are all handled by separate classes. This makes the system easier to extend or modify without affecting unrelated components. Rather than the ETLPipeline performing file operations and business logic directly, it delegates those tasks to specialized classes. This creates a cleaner architecture where each component has a well-defined role, making the overall design more modular and easier to understand.
